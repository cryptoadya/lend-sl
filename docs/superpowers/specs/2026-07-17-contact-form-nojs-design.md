# Надёжная отправка контактной формы без JavaScript

## Цель

Реальный пользователь должен иметь возможность отправить заявку как при обычной работе JavaScript, так и при его отключении или сбое. В production обе разновидности заявки должны проходить через существующий Resend-путь и приходить на тот же e-mail. Preview-окружение `dev` по-прежнему использует `CONTACT_MODE=mock` и не отправляет реальные письма.

## Выбранное решение

Форма получает явные `method="post"` и `action="/api/contact"`. При работающем JavaScript существующий обработчик перехватывает отправку, передаёт JSON и показывает результат внутри страницы. Без JavaScript браузер отправляет обычный `application/x-www-form-urlencoded` POST на тот же endpoint.

API принимает только два формата: JSON от клиентского скрипта и URL-encoded данные от браузера. После разбора оба формата преобразуются в одну внутреннюю структуру и проходят одну и ту же серверную валидацию, honeypot и доставку через Resend.

Атрибут `novalidate` удаляется. Благодаря этому браузер без JavaScript проверяет обязательные поля, формат e-mail и согласие с политикой конфиденциальности до отправки. Условие «телефон или e-mail» остаётся дополнительно защищено общей серверной проверкой.

## Ответ пользователю

Для JSON API продолжает возвращать нынешние JSON-ответы и HTTP-статусы.

Для обычной HTML-формы:

- после успешной отправки сервер отвечает перенаправлением `303` на статическую страницу подтверждения;
- honeypot получает такое же внешне успешное перенаправление, но письмо не отправляется;
- при ошибке проверки или временной ошибке доставки сервер перенаправляет на отдельную безопасную страницу ошибки с предложением вернуться к форме либо связаться по телефону/e-mail;
- введённые имя, контакты и сообщение не включаются ни в URL, ни в HTML-ответ, ни в логи.

Перенаправление после POST исключает повторную отправку заявки при обновлении страницы.

## Безопасность и антиспам

- Принимаются только `application/json` и `application/x-www-form-urlencoded`; остальные типы отклоняются.
- Сохраняются существующие ограничения размера запроса и длины полей.
- Для обоих форматов применяется одна серверная проверка имени, сообщения, обратного контакта, e-mail и согласия.
- Honeypot `website` действует одинаково для JSON и HTML-формы.
- Production отправляет письмо только при `CONTACT_MODE=resend` и корректной конфигурации Resend.
- Preview остаётся в `CONTACT_MODE=mock`, поэтому тестирование не создаёт реальных писем.
- Будущий rate limiting совместим с обоими режимами. Если Turnstile станет обязательным, для пользователей без JavaScript понадобится отдельное решение, потому что интерактивный Turnstile обычно зависит от JavaScript.

## Область изменений

- `src/components/ContactCTA.astro`: явный POST, стандартная браузерная валидация.
- `functions/api/contact.ts`: безопасный разбор двух форматов, единая обработка, JSON- или redirect-ответ в зависимости от типа отправки.
- Статические страницы успешного и неуспешного результата без персональных данных.
- Автоматические тесты формы и API.
- `docs/audit-sline/README.md`: отметка `FORM-NOJS` после проверки dev Preview.

## Проверка результата

1. Автотесты доказывают, что JSON-сценарий не изменился.
2. Автотесты проверяют URL-encoded запросы: успех, валидацию, honeypot и отсутствие отправки в mock-режиме.
3. Сборка проекта проходит без ошибок.
4. В браузере с JavaScript форма отправляется в dev Preview и показывает обычное сообщение об успехе.
5. В браузере с отключённым JavaScript форма отправляется POST-запросом, не добавляет данные в URL и открывает страницу подтверждения.
6. Основной сайт и ветка `main` не изменяются до отдельного решения владельца.
